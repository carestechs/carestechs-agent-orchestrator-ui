### T-004: BFF session middleware and `/auth/*` routes

**Type:** Backend
**Workflow:** standard
**Complexity:** M
**Dependencies:** T-001

**Description:**
Implement cookie-based session middleware (httpOnly, sameSite=lax, secure in prod) and the three auth endpoints: `POST /auth/login` (passphrase compare against `ORCHESTRATOR_OPERATOR_PASSPHRASE`), `POST /auth/logout` (clear cookie, 204), `GET /auth/me` (returns `{ authenticated, expiresAt }`).

**Rationale:**
The SPA never holds the orchestrator key; the BFF session is the only credential the browser sees. Required by AC "no Authorization header in browser network logs".

**Acceptance Criteria:**
- [ ] Login compares passphrase via constant-time compare; returns `401 invalid-passphrase` on mismatch (RFC 7807).
- [ ] Successful login sets a signed session cookie with TTL ≥ 8h and returns `expiresAt`.
- [ ] `GET /auth/me` returns `{ data: { authenticated: false } }` with HTTP 200 when no/expired session — used by the auth guard.
- [ ] Cookie is `httpOnly`, `sameSite=lax`, and `secure: true` when `NODE_ENV=production`.
- [ ] Unit tests cover login success, login failure, logout, expired session, and cookie attribute assertions.

**Files to Modify/Create:**
- `bff/src/session/cookie-session.ts` — sign/verify logic.
- `bff/src/routes/auth.ts` — login/logout/me handlers.
- `bff/src/server.ts` — register middleware and routes.
- `bff/src/routes/auth.spec.ts`.

**Technical Notes:**
- Use a session secret env var (`SESSION_SECRET`); fail fast on boot if unset in production.
- Do not log passphrase or session secret values.

### T-005: BFF `/api/v1/*` JSON pass-through with bearer injection

**Type:** Backend
**Workflow:** standard
**Complexity:** M
**Dependencies:** T-004

**Description:**
Implement a generic forwarder that proxies all `/api/v1/*` requests to `${ORCHESTRATOR_BASE_URL}/v1/*`, injects `Authorization: Bearer ${ORCHESTRATOR_API_KEY}`, requires a valid session, and passes through status, headers, and `application/problem+json` bodies unchanged.

**Rationale:**
Single chokepoint guarantees the API key never reaches the browser and that error envelopes stay RFC 7807-compliant per `docs/api-spec.md`.

**Acceptance Criteria:**
- [ ] Unauthenticated request → `401 unauthenticated` (problem+json) without contacting upstream.
- [ ] Authenticated request to `GET /api/v1/runs` returns the upstream payload byte-identical (modulo hop-by-hop headers).
- [ ] Upstream `409`/`404`/`422` problem+json responses are passed through unchanged with their `code` field intact.
- [ ] Connection failure to orchestrator → `500 upstream-unavailable` problem+json; upstream 5xx → `502 upstream-error`.
- [ ] Logs never contain the value of `ORCHESTRATOR_API_KEY` or `Authorization`.

**Files to Modify/Create:**
- `bff/src/upstream/orchestrator-client.ts` — fetch wrapper, header injection.
- `bff/src/routes/api-proxy.ts` — generic forwarder for JSON endpoints.
- `bff/src/upstream/orchestrator-client.spec.ts`.

**Technical Notes:**
- Use Node's native `fetch`; do not use libraries that auto-decompress (it complicates streaming in T-006).
- Strip `host`, `connection`, `transfer-encoding` from forwarded request headers.

### T-006: BFF NDJSON streaming pass-through for `/api/v1/runs/:id/trace`

**Type:** Backend
**Workflow:** standard
**Complexity:** M
**Dependencies:** T-005

**Description:**
Add a dedicated route that opens an upstream `fetch` to `/v1/runs/:id/trace` (with `follow`, `since`, `kind` query forwarded) and pipes the response body straight to the client using `pipeline()` — no buffering, no JSON parsing, no compression in the middle.

**Rationale:**
The trace is the live signal-arrival channel; any buffering kills the AC "trace begins streaming within 1s" and breaks `meta.alreadyReceived` detection. Identified as a risk in the feature brief.

**Acceptance Criteria:**
- [ ] Response `Content-Type` is `application/x-ndjson` and `Transfer-Encoding: chunked`.
- [ ] Lines arrive at the client incrementally — verified by an integration test that reads the first line in <500ms while upstream stalls between lines.
- [ ] Client disconnect aborts the upstream fetch (`AbortController`).
- [ ] Upstream `401` on the trace stream propagates as `401 unauthenticated` to the SPA without buffering.
- [ ] No `Content-Encoding: gzip|br` is set on the response; any upstream encoding is decoded server-side or disabled.

**Files to Modify/Create:**
- `bff/src/routes/trace.ts`.
- `bff/src/upstream/orchestrator-client.ts` — add `streamTrace(runId, query)`.
- `bff/src/routes/trace.spec.ts` — uses an upstream mock that yields lines with delays.

**Technical Notes:**
- Disable any reverse-proxy buffering downstream via `X-Accel-Buffering: no`.
- Use `pipeline(upstream.body, reply.raw)` (Fastify) or `upstream.body.pipe(res)` (Express); flush headers immediately.

### T-010: TraceStreamService — `fetch` + `ReadableStream` to signals

**Type:** Frontend
**Workflow:** standard
**Complexity:** L
**Dependencies:** T-003

**Description:**
Implement a service that opens `GET /api/v1/runs/:id/trace?follow=true` via `fetch`, reads the body as a `ReadableStream<Uint8Array>`, splits it on `\n` boundaries, JSON-parses each line into a `TraceRecord`, and exposes the rolling list as a signal. Handle reconnection on transient network errors using `since=<lastTimestamp>` to resume.

**Rationale:**
Pure RxJS doesn't ergonomically read NDJSON; per `CLAUDE.md`, fetch+ReadableStream is the prescribed shape, bridged into a signal for the component.

**Acceptance Criteria:**
- [ ] Records appear in the signal within 1s of upstream emission (verified with mocked stream).
- [ ] Partial-line buffering across chunk boundaries is correct (test with chunks split mid-record).
- [ ] On `AbortController.abort()` the fetch closes and no further records are emitted.
- [ ] On network drop with `follow=true`, the service reconnects with `since=<lastRecordTimestamp>` once and gives up after N retries with an error signal.
- [ ] On `401`, the service stops and emits an auth-expired event consumed by the guard.

**Files to Modify/Create:**
- `src/app/core/trace-stream.service.ts`.
- `src/app/core/trace-stream.service.spec.ts` — uses `ReadableStream` polyfill or `whatwg-streams` testing helpers.

**Technical Notes:**
- Do **not** use `EventSource` or WebSocket; this is plain NDJSON over HTTP per `CLAUDE.md` anti-patterns.
- Keep the parser pure (string in, `TraceRecord[]` out) so it's unit-testable without a server.
